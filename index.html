Automated change analysis of the AIUC-1 standard across its quarterly releases. Numbers below are derived at runtime from the source repository and reconciled against the published website narrative. Snapshots are anchored to git commits because the source publishes no version tags.

Methodology

Three analysis layers, strictly isolated (v03 §6):

  • 6a Requirement-level (authoritative). Per-ID added/removed/modified; modified graded into editorial / clarification / substantive via normalised edit distance.
  • 6b Should/may composition (authoritative, exact). Counts and the should:may ratio per requirement and per principle. Interpretation as obligation hardening/softening is inference, not a count fact.
  • 6c Per-bullet matching (release-type-gated, confidence-banded, isolated). Greedy best-match over the combined should+may pool with merge-flag heuristic. Indicative only on structural releases.

There is no native change stamp in the AIUC-1 source. Nature classification is therefore distance-driven and currently provisional: the website narrative grades requirement+control changes jointly, while 6a grades requirement-body text only, so the narrative is not a clean calibration anchor for the body-text thresholds.

Sources and use
